What is Version Control?

Version Control System (VCS): Software that records changes to files so that you can:

  • Revert to a previous state
  • Compare changes over time
  • See who made changes

 

No more catastrophes!

What is Git?

 

Git: Free, open source version control system

  • Command line application
  • Runs on your local machine

What is GitHub?

 

GitHub: Code storage + sharing service based on Git that allows you to:

  • Collaborate with others on projects
  • Share/showcase your code
  • Find code that you can use in your own projects

Git Terms

  • Repository ("Repo"): a project (folder) that Git is tracking changes to
  • Remote Repository: remote copy of a Git repository (ie: in GitHub)
  • Working Directory: local copy of a Git repository
  • Commit: a change set recorded in Git
  • Index: staging area - list of changes not yet committed
  • Head: list of committed changes

Git workflow

Let's Git going!

Create a GitHub account

https://github.com

Create a new GitHub repository

  1. Click + in the upper right and choose New Repository
  2. Fill out the form and click Create Repository (see example on next slide)

Setting up Git

Download + install Git

https://git-scm.com/downloads

Check that Git is installed

git --version

Setting up Git

Add your name
(doesn't need to match your GitHub username)

git config --global user.name "YOUR NAME"

 

And the email attached to your GitHub account

git config --global user.email "YOUR EMAIL"

Clone your repo

Find the URL for your GitHub repo

Clone your repo

Clone it your local machine

git clone https://github.com...

Change directory into your local repo

cd your-repo-name

What's in there?

ls -la

Making changes

Edit README.md and save your changes

vim README.md

(or open it in a text editor)

Making changes

git status
Check to see what changes Git has noticed

git status
On branch master
Your branch is up-to-date with 'origin/master'.
Changes not staged for commit:
  (use "git add ..." to update what will be committed)
  (use "git checkout -- ..." to discard changes in working directory)

    modified:   README.md

Making changes

git add
Adds files to the Index (staging area for your next commit)

 

Add all changed files

git add -A

Add a single file

git add README.md

Making changes

git add
ed something you didn't intend to?

git reset
Removes files from the Index

 

Remove all files

git reset

Remove a single file

git reset README.md

Making changes

git commit
Commits your changes to the Head

  • A commit is a waypoint in your project that you can refer/revert back to
  • Every commit requires a "commit message" - a note about the purpose of the change

 

git commit -m "added README file"

Making changes

Push your local commits to your remote repo

git push origin master

GitHub will ask for your password whenever you push to your remote. You can cache your password, create a personal access token or connect with SSH

Revert changes

Edit README.md again and save

# demo-repo
This is a terrible change

Add and commit the change

git add README.md
git commit -m "this was a terrible idea"

Revert changes

git log

Revert changes

Revert to a specific commit

git reset 603c619cf90dd9520c141d65d0f3697973bc9a8d

Things have gone terribly wrong?

Reset local to match remote

git reset --hard origin master

Playing nicely with others

Default: Everyone works on, commits and pushes to the master branch

  • OK if you're working alone
  • Not great if you're working with others - you end up overwriting each others' changes

Playing nicely with others

Git allows users to collaborate through 2 features:

  • Branching allows each user to work on a separate copy of a repo
  • Merging allows combining branches and checking for conflicts

Branching workflow

Branching


Create a new local branch

git checkout -b new-branch-name

Branching


Make and commit some changes

vim README.md
git add README.md
git commit -m "editing that readme again"

Branching

Push your local changes to a new branch in your remote repo

git push origin new-branch-name

Merging

After a branch is pushed to the remote, it needs to be checked for conflicts and merged into the master branch to make the changes 'official'



In GitHub, request to merge your branch into master by making a Pull Request

Pull requests

Pull requests send a message to the remote repository owner asking to merge your branch into the master branch

  • Specific to GitHub (not part of Git)
  • Can only be made via the GitHub UI

Create a pull request

Switch to your branch

Click New Pull Request

Merge a pull request

If there are no conflicts, a pull request can be merged with one click

Merging your own Pull Requests? Not a great practice.
Always get someone else to review if you can!

Syncing

After pull requests are merged, always sync your local repo with the remote.

Switch to the master branch

git checkout master

Pull the lastest code from the remote repo

git pull origin

Syncing

If you're still working on a local branch, you need to keep your branch up to date by regularly merging the latest master

Switch to your local branch

git merge master
